## LinkedIn

**Character count: 1,139/1,300**

The naive LLM failover — try OpenAI, catch, call Anthropic — works in dev and breaks three ways under production load.

Retry storms: every failed request hits the struggling API before falling back, doubling the load at 1K concurrent. No health memory: each request independently discovers the provider is down, paying 2-5s timeout latency every time. Error blindness: a 429 means "slow down," a 503 means "go elsewhere," a 400 means "the payload is broken" — treating them the same cascades failures.

The fix is a FailoverRouter with error classification, health-tracked cooldowns, and a sliding window that auto-triggers failover when failure rate crosses a threshold.

Benchmarked the implementation. Happy-path overhead: ~3μs at p50. Daily cost increase: +1.1% at any scale. The surprising finding: cooldown threshold sensitivity — at 0.8, users absorb 1,335 failover events vs. 29 at 0.2.

Full pattern with TS + Python code, 6 benchmark scenarios, cost analysis, and operational runbook:
